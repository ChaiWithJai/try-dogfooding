# ADR 002: Dogfood CLI v0.1.0 Architecture & Decisions

**Date:** 2026-04-25  
**Status:** Accepted  
**Context:** Release of the v0.1.0 Dogfood CLI, representing the first functional slice of the product for workshop cohorts.

---

## 1. Context and Scope

The v0.1.0 release establishes the foundational mechanics for Dogfood operators to run agentic software against their own data. It introduces:
- A command-line interface (`@trydogfooding/cli`) built in TypeScript for Node.js.
- Workspace scaffolding with a strict Git-backed file structure.
- Declarative workflow schemas (`workflow.yaml`) validated by Zod.
- Execution guardrails wrapped around a Claude Code subprocess.
- Append-only JSONL run history for auditing.
- The `cx-triage` bundled workflow template.

This document serves as the historical record for the architectural decisions, tradeoffs, missteps, and constraints that shaped this initial release.

---

## 2. Core Architectural Decisions

### 2.1 Subprocess wrapping vs Anthropic Agent SDK
**Decision:** We run Claude Code via `child_process.spawn("claude", ...)` rather than using the in-process Agent SDK.
**Rationale:** The Agent SDK is evolving, whereas the `claude` CLI is battle-tested. Using a subprocess allowed us to immediately ship a stable execution engine with known behaviors, avoiding breaking changes during early cohort testing.

### 2.2 Append-Only JSONL History
**Decision:** Run logs are stored in `history/runs.jsonl`. No SQLite, no local database.
**Rationale:** JSONL is universally parseable, Git-friendly, and completely transparent to the operator. It requires zero initialization steps or database migrations, perfectly aligning with the "local files only" ethos.

### 2.3 Strict Schema Validation (Zod)
**Decision:** `workflow.yaml` and `config.yaml` are strictly validated at runtime before prompt assembly begins.
**Rationale:** Operators are not developers. YAML indentation errors or missing keys cause opaque failures later in execution. Zod allows us to catch configuration errors instantly and provide actionable, human-readable `next-steps` via the `DogfoodError` system.

### 2.4 Commander + Ora vs Ink TUI
**Decision:** We built standard terminal output using Chalk and Ora, intentionally deferring a React-based TUI (like Ink).
**Rationale:** TUI frameworks add dependency weight and complexity to testing. Standard stdout/stderr streams cleanly support the `--json` flag out-of-the-box, ensuring scriptability and headless CI/CD integrations for advanced users.

## 3. Tradeoffs Made

### 3.1 Speed to Market vs Deep Integration
By utilizing a subprocess, we surrendered fine-grained event streaming. We cannot capture real-time tool usage or intermediate thoughts seamlessly—we only capture the final `stdout` and execution duration. This tradeoff was accepted to get a functional product into the hands of the first cohort immediately.

### 3.2 Strict Guardrails vs Ad-hoc Flexibility
We enforce strict timeouts (default 120s), token budgets, and full-jitter retries wrapped around every Claude call. We sacrifice the operator's ability to run infinite loops or unbounded explorations, prioritizing safety and billing limits over absolute flexibility.

## 4. Wrong Ideas and Why They Failed

During the development and testing of v0.1.0, several initial assumptions proved functionally or architecturally broken and had to be rewritten.

### 4.1 Wrong Idea 1: Sandboxing Claude to the Workflow Directory
**The Idea:** To keep workflows isolated, we initially set the Claude Code subprocess `cwd` (Current Working Directory) to `workflows/<workflow-name>/`.
**Why it failed:** The operator's source data resides at the workspace root (`data/`). Because Claude Code's security model requires interactive `y/n` confirmation to read files outside its sandbox—and because Dogfood runs the subprocess in a non-interactive pipe (`-p`)—Claude silently timed out or dropped the file read operations, resulting in empty reports.
**The Fix:** We elevated the subprocess `cwd` to the workspace root, and injected relative path resolution into the prompt assembly to ensure outputs were still written correctly to the workflow folders.

### 4.2 Wrong Idea 2: Throwing early on execution failure
**The Idea:** If `executeWithClaude` encountered a fatal error (e.g., binary missing, timeout), it threw an Error immediately, bubbling up to the global catch block.
**Why it failed:** It completely bypassed the `appendRun` logging step. Failed runs left zero audit trail in `history/runs.jsonl`, violating the spec's guarantee that *every* run produces an auditable record.
**The Fix:** We wrapped the execution step in an inner `try/catch`, forcing the CLI to synthesize a failed `RunRecord`, append it to the log, and *then* re-throw to exit gracefully.

### 4.3 Wrong Idea 3: The CLI silently installing Claude Code
**The Idea:** If `dogfood init` didn't find Claude Code, it would transparently run `npm install -g @anthropic-ai/claude-code`.
**Why it failed:** It violated the product rule of transparent system mutation. Global installations can require `sudo`, modify paths unexpectedly, or break existing setups. 
**The Fix:** We pivoted to a diagnostic approach (`dogfood doctor`), detecting the absence and providing explicit instructional fallbacks: `"Install Claude Code: https://claude.ai/code"`.

## 5. Security & Legal Constraints

- **AGPLv3 Integrity:** The CLI strictly adheres to the AGPL-3.0-or-later license. There are zero "commercial-only" CLI features or closed-source tracking binaries.
- **Zero Credential Storage:** The CLI **never** stores or requests API keys. All authentication is delegated entirely to the Claude Code credential manager. `config.yaml` is sterile.
- **No Default Telemetry:** `telemetry_enabled: false` is permanently hardcoded as the default initialization state.

## 6. Marketing Promises vs DX Reality

A core tenet of Dogfood is building exactly what we market. Here is how v0.1.0 aligned with our public promises, and where Developer Experience (DX) drift was identified.

- **Promise:** *"Software runs against their real data."*
  - **Reality:** Achieved. By fixing the data sandboxing bug (Section 4.1), templates assemble prompts accurately against the user's root `data/` directory.
- **Promise:** *"Setup failures are our fault."*
  - **Reality:** Achieved. We implemented the `DogfoodError` system to ensure *every* failure includes a clear `next-step`. Raw stack traces are never exposed unless `--verbose` is used.
- **Promise:** *"The happy path is one command."*
  - **Reality:** **DX Drift.** The CLI requires the operator to manually install and authenticate Claude Code if it is missing. For a non-technical operator, installing global npm packages is high-friction. We documented this drift and accepted it for v0.1.0, but v0.2.0 will require migrating to the in-process Agent SDK to eliminate this external dependency friction.
